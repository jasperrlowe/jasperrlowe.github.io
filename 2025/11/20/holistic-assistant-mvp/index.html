全人助手 ②：MVP — 用 AI 整合培养方案 PDF 做学业规划

TL;DR: 花了 3 周做出”全人助手”的第一个可用版本，验证了用 AI 整合培养方案 PDF 来做学业规划这件事是可行的。227 行 Flask 后端 + 1189 行原生 JS 前端，支持 36 个专业的课程查询。

🎯 MVP 要验证什么

做之前有三个核心问题：

  1. 学生真的需要这样一个工具吗？
  2. Gemini 能理解中文培养方案 PDF 并给出靠谱回答吗？
  3. 不爬 SIS、不建数据库，能不能做出可用的东西？

做了什么、没做什么

完整版想法 MVP 做法 为什么
爬 SIS 实时数据 ❌ 只用 PDF 先验证 AI 能不能理解 PDF
Vector DB 语义检索 ❌ 直接读文件 + 内置模板 1500 门课不需要数据库
React 前端 ❌ 原生 JS + Tailwind 快速迭代，不需要构建工具
用户登录 ❌ localStorage MVP 不需要后端状态

结果：3 周开发 + 2 周内测，三个假设都验证通过了。

1. Hybrid Context：不用 Vector DB 的 RAG

  • “接入了知识库的AI回答比直接上网搜索靠谱”（8/12）

  • “规划功能太弱”（5/12）→ 后来加了可视化路线图

⚠️ MVP 的核心缺陷：AI 输出不受控

MVP 阶段最大的问题其实不是技术栈选型，而是 AI 的回答太不受约束了。

理想中的效果是：用户问”CS 大二应该选什么课”，系统基于培养方案给出结构化的推荐。但实际情况是，AI 经常自由发挥：

  • 编造不存在的课程：比如培养方案里没有”CSC2050”，但 AI 会自信地推荐它，还编一段课程描述
  • 忽略约束条件：用户说”我只剩 12 学分”，AI 推荐了 18 学分的课表，完全无视限制
  • 风格不一致：同一个问题问两遍，回答格式、详细程度差异很大，有时是表格，有时是大段文字
  • 过度”热情”：经常给出与问题无关的建议，比如问学分要求，它还会”顺便”规划整个大学四年

简单来说，MVP 阶段的 AI 更像是一个”什么都敢说的热心学长”，而不是一个可靠的学业顾问。这和理想中那种”精确、有据可依、格式统一”的效果出入很大。

这个问题直接推动了 v2 的两个改进方向：

  1. 用 Hybrid Context 注入官方数据，让 AI 只基于给定事实回答
  2. 在 Prompt 里加硬性约束（”If data is missing, say 官方数据未提供”），强制 AI 承认不知道

🔍 技术债和 v2 方向

问题 原因 v2 怎么解决
2 份 PDF 解析失败 PyPDF2 处理不了扫描版 换 Gemini Vision API
没有开课信息 PDF 只有课程列表 Playwright 爬 SIS
每次喂整个 PDF（50K tokens） 没做按需加载 按课程分片存 JSON
硬编码模板难维护 培养方案改了就要手动改代码 从 PDF 自动提取

🎓 我学到了什么

技术上：

  • RAG 不一定要上 Vector DB，数据量小的时候文件系统就够了
  • 简单的关键词路由比用 AI 做意图分类快 10 倍，MVP 阶段够用
  • 原生 JS + Tailwind 做原型很快，不需要构建步骤

产品上：
用户真正想要的不是聊天机器人，而是能点击的决策工具：

“我不想打字问’CSC3100 先修是什么’，我想在课程表上点一下就能看到”

这直接影响了 v2 的方向：从纯聊天变成可视化路线图 + 课程可点击。

📚 技术栈

层级 技术 选择理由

后端 Flask 最简单的 Python API 框架
LLM Gemini Flash + Pro Flash 快，Pro 准
PDF 解析 PyPDF2 轻量，文本型 PDF 够用
