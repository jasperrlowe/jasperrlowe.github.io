全人助手 ③：v2 — 从 PDF 到 1568 门课程的 RAG 学业顾问系统

TL;DR: 在 MVP 的基础上，把系统从”只能读 PDF”升级成了完整的学业顾问：加了 SIS 爬虫（1568 门课 × 3 学期），用 Gemini 做 PDF 去噪，按学期分片存储来管理上下文。核心挑战：PDF 数据提取的噪声过滤、SIS 反爬、上下文窗口管理。

📌 要解决的问题

CUHK-SZ 学生选课时要对付两个互不相通的数据源：

  1. 培养方案 PDF（36 份，每份 30-50 页）：有课程列表，但没有学分、先修要求、开课时间
  2. SIS 系统：有开课详情，但只显示当前学期，没有 API，需要手动一门门搜

结果就是：学生想知道”CSC3100 什么时候开、先修是什么”，要同时翻 PDF 和 SIS，开 10 多个标签页对比，平均耗时 45 分钟。

目标：把这两个数据源整合到一起，用 AI 做自然语言问答，响应时间 <2s。

🔧 三个核心技术难题

1. PDF 数据提取：正则 + AI 去噪

直接用正则 \b([A-Z]{2,5})\s?(\d{4})\b 提取课程代号，误报率 40%——会把日期（APRIL2024）、学期词（SPRING）也当成课程。

SIS 用的 PeopleSoft，反爬挺严格：session 15 分钟过期，请求太快会弹验证码。

3. 上下文窗口管理：按学期分片

如果每次都把整个 PDF（50K tokens）喂给模型，几轮对话就会超窗口限制。

🔍 做完之后的反思

数据质量 > 模型能力

Gemini Flash + 干净数据 比 GPT-4 + 噪声数据 效果好得多。在课程先修查询任务里，GPT-4（无 RAG）准确率 62%，Gemini Flash（Hybrid Context）96.8%。与其换更强的模型，不如先把数据清洗好。

显式的错误处理

早期版本碰到知识库里没有的课程，AI 会瞎编答案。后来改成显式返回”数据库暂无此课程信息”，附上可能原因和建议。用户反而更信任系统了。

只用官方数据

有人建议接入课程评价（类似”卡园”），但 UGC 有法律风险和质量问题（”水课”定义因人而异），暂时只用培养方案和 SIS 这两个官方数据源。

⚠️ v2 仍然存在的问题

虽然 v2 在数据覆盖和准确率上有了明显提升，但 RAG 系统在实际使用中还是暴露了一些没解决好的问题：

相似课程代号分不清

最典型的例子是 STA2001 和 STA2001H——一个是普通版，一个是荣誉版（Honor），内容和要求完全不同。但系统在检索和回答时经常把它们混在一起，比如：

  • 用户问 STA2001H 的先修要求，系统返回的是 STA2001 的信息
  • 在生成学业规划时，两门课被当成同一门，导致推荐重复

根本原因是正则提取和精确匹配都是按”前缀 + 数字”来识别的，后面的字母后缀（H、A、B 等）在匹配逻辑里没有被严格区分。这类问题在有 Honor Track 的专业（如 Statistics、Math）里尤其明显。

中文描述被误识别为课程

培养方案 PDF 里经常出现类似”三个选修”、”两门核心课”这样的中文描述。AI 在处理 PDF 上下文时，偶尔会把这些描述性文字当成具体的课程名，直接加入生成的规划里。比如：

用户：”帮我列出大二要上的课”
AI 输出：”… CSC3100、MAT2040、三个选修、PHY1002 …”

“三个选修”不是一门课，而是一个要求说明。但因为 PDF 的排版不规律（有时课程名和描述混在同一行），系统没法可靠地区分”这是课程名”还是”这是一段说明文字”。

这两个问题本质上都是 RAG 的粒度不够细——系统能做到”找到相关文档”，但在”精确理解文档内容”这一步还差得远。后续可能需要：

  • 对课程代号做更严格的标准化（包括后缀）
  • 在 PDF 提取阶段就区分”课程实体”和”描述文本”，而不是交给 LLM 自己判断

🎓 作为大一学生学到的东西

技术成长：

  • 之前以为 RAG 就等于 Vector DB，做完才知道结构化数据用文件系统 + 精确匹配就够了
  • 第一次在真实场景处理反爬，学会了 session 管理和节流策略
  • 在 Windows 上搞 multiprocessing 踩了不少坑，理解了 pickling 的限制

做项目的体会：

  • MVP 不是偷懒，是快速验证假设——与其花 3 个月做完美爬虫，不如 2 周验证”学生是不是真需要这个”
  • 用户更在乎时效性（”这学期开不开”）而非完整性（”历史上开过几次”），这影响了数据优先级

📚 技术栈

层级 技术 选择理由
前端 Vanilla JS + Tailwind 轻量，不需要构建工具
后端 Flask 简单的 Python REST API
LLM Gemini 2.5 Flash 便宜，响应快
爬虫 Playwright 比 Selenium 轻量，支持 headless
数据处理 PyPDF2 + 正则 文本型 PDF 不需要 OCR
知识库 本地 JSON 文件 <2K 文档不需要数据库
