<p>最近有点丧。</p>
<p>这几个月 AI Agent 和 Skill 的发展速度太快了——Gemini、Claude 这些模型已经能通过工具调用直接访问网页、读文件、执行代码。各种 MCP 协议、Agent Framework 层出不穷，感觉只要写一个 skill 接进去，AI 就能自己去 SIS 查课、读培养方案、帮你排课表。</p>
<p>那我花了几个月做的全人助手，是不是其实不如直接写成一个 skill，接进现成的 Agent 框架里就行了？</p>
<span id="more"></span>
<hr/>
<h2 id="😔-被”新技术”冲击的感觉"><a class="headerlink" href="#😔-被”新技术”冲击的感觉" title="😔 被”新技术”冲击的感觉"></a>😔 被”新技术”冲击的感觉</h2><p>说实话，第一次看到有人用 Agent + Skill 几行配置就搞定了一个类似功能的 demo，心里挺不是滋味的。我辛辛苦苦写爬虫、搞 PDF 解析、调 prompt、做前端，人家可能一个下午就能搭出差不多的东西。</p>
<p>而且从架构上看，Agent 的做法确实更”优雅”：</p>
<ul>
<li>不需要预先爬取和存储数据，Agent 实时去查就行</li>
<li>不需要维护知识库，模型自己决定什么时候查什么数据</li>
<li>不需要做上下文管理，Agent 框架帮你处理</li>
</ul>
<p>这让我一度觉得自己的 RAG 方案是在”用旧方法解决新问题”。</p>
<hr/>
<h2 id="😵-RAG-本身的困境：离散-vs-连续"><a class="headerlink" href="#😵-RAG-本身的困境：离散-vs-连续" title="😵 RAG 本身的困境：离散 vs. 连续"></a>😵 RAG 本身的困境：离散 vs. 连续</h2><p>除了被 Agent 冲击之外，RAG 本身的问题也一直困扰着我。</p>
<p>核心矛盾在于：<strong>SIS 的数据是离散的，PDF 的数据是连续的，但 AI 需要把它们当成统一的知识来理解。</strong></p>
<p>SIS 里的课程信息天然是结构化的碎片——每门课是一条独立记录，有课程代号、学分、开课时间、老师，彼此之间没有叙述上的连贯性。你可以精确地查到 CSC3100 在 2025-26 第一学期开了 3 个 section，但这些信息是”离散的点”。</p>
<p>而培养方案 PDF 恰好相反——它是连续的叙述：从大一到大四的课程安排有逻辑顺序、有先修依赖、有模块分组，课程之间有上下文关系。但 PDF 一旦被提取成文本，这些结构关系就很容易丢失。</p>
<p>问题就在这里：<strong>AI 擅长理解连续文本，但对离散碎片的整合能力有限。</strong> 你喂给它一堆 JSON 格式的课程数据，它能读懂每条记录，但很难自动理解”CSC1001 → CSC1002 → CSC3100”这条依赖链——因为这个关系存在于 PDF 的连续叙述中，而不是 SIS 的离散字段里。</p>
<p>反过来，如果你把整个 PDF 喂给它，它能理解课程体系的全貌，但又会在具体细节上犯错（比如分不清 STA2001 和 STA2001H）。</p>
<p>这种”离散”和”连续”的张力，是我目前 RAG 方案最大的瓶颈。</p>
<hr/>
<h2 id="💪-但是，这个项目的价值还在"><a class="headerlink" href="#💪-但是，这个项目的价值还在" title="💪 但是，这个项目的价值还在"></a>💪 但是，这个项目的价值还在</h2><p>想了很久之后，我觉得这个项目还是有做下去的意义。原因很简单：</p>
<p><strong>全人助手里的数据，是网上搜不到的。</strong></p>
<p>你去 Google 搜”CUHK-SZ CSC3100 先修要求”，大概率搜不到准确结果。SIS 是校内系统、需要登录才能访问，培养方案 PDF 散落在各学院官网的角落里，而且每年都在更新。这些信息不在任何公开知识库里，ChatGPT 和 Gemini 的训练数据里也没有。</p>
<p>就算 Agent 技术再成熟，它也需要有人告诉它”去哪里找数据、怎么登录 SIS、怎么理解这份 PDF 的结构”。而这正是我这几个月做的事情——<strong>把这些分散的、私有的、非标准化的数据变成可用的知识库</strong>。</p>
<p>Agent 是工具，但知识库是资产。工具可以被替代，资产不容易被替代。</p>
<hr/>
<h2 id="🔭-接下来怎么办"><a class="headerlink" href="#🔭-接下来怎么办" title="🔭 接下来怎么办"></a>🔭 接下来怎么办</h2><p>与其纠结”要不要重写成 Agent skill”，不如把精力放在这个项目真正有壁垒的地方：<strong>数据质量和个性化算法</strong>。</p>
<p>接下来想重点做的事情：</p>
<ul>
<li><strong>个性化推荐算法</strong>：不只是”查课程信息”，而是根据学生的专业、年级、已修课程、GPA 等，给出真正个性化的选课建议。这不是通用 Agent 能做到的，因为它需要对 CUHK-SZ 培养方案的深度理解</li>
<li><strong>课程关系图谱</strong>：把 PDF 里隐含的先修依赖、模块归属关系抽取出来，建成结构化的图谱，解决”离散 vs. 连续”的问题</li>
<li><strong>历史数据积累</strong>：持续爬取多学期的 SIS 数据，积累开课规律（比如”这门课每年只在 Fall 开”），这些是实时查询拿不到的</li>
<li><strong>用户反馈闭环</strong>：让用户标记”这个建议有没有用”，逐步优化推荐的准确度</li>
</ul>
<p>技术栈可以变、架构可以改，但只要数据和算法是独有的，这个项目就有存在的价值。</p>
<hr/>
<p>虽然现在有点迷茫，但也不至于放弃。先把能做的做好，边做边想。</p>
